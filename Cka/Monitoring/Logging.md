# Kubernetes Logging — Notes

# Overview

Kubernetes provides basic logging functionality to:

* View application logs
* Stream live logs
* Troubleshoot containers and pods

Logs are usually generated by applications and written to:

```text id="x11q2s"
STDOUT / STDERR
```

Kubernetes captures these logs and makes them accessible using `kubectl logs`.

---

# Logging in Docker

## Example Scenario

A Docker container called:

```text id="u8m4kp"
event-simulator
```

generates random events simulating a web server.

These logs are printed to standard output.

---

## Running a Container in Detached Mode

```bash id="z0v9nb"
docker run -d <image>
```

* `-d` runs the container in background mode.
* Logs are not directly visible.

---

## Viewing Docker Logs

```bash id="m3n5xq"
docker logs <container-id>
```

---

## Stream Live Logs

```bash id="j8t2ra"
docker logs -f <container-id>
```

### `-f` option

```text id="y7d0vl"
follow
```

Streams logs continuously in real time.

---

# Logging in Kubernetes

Kubernetes provides similar logging functionality through:

```bash id="t9u4ge"
kubectl logs
```

---

# Viewing Pod Logs

## Basic Syntax

```bash id="r3k8cw"
kubectl logs <pod-name>
```

Example:

```bash id="n4p2sx"
kubectl logs event-simulator-pod
```

Displays logs from the container inside the pod.

---

# Stream Live Logs

```bash id="w1x6oe"
kubectl logs -f <pod-name>
```

This continuously streams logs in real time.

---

# Multi-Container Pods

## Important Concept

A Kubernetes pod can contain:

* One container
* Multiple containers

Example containers:

* `event-simulator`
* `image-processor`

---

# Problem with Multiple Containers

If a pod contains multiple containers:

```bash id="p2m5vz"
kubectl logs <pod-name>
```

may fail because Kubernetes does not know which container logs to display.

---

# Specify Container Name

Use:

```bash id="c5k9ut"
kubectl logs <pod-name> <container-name>
```

Example:

```bash id="e8r1fa"
kubectl logs event-simulator-pod event-simulator
```

---

# Stream Logs from Specific Container

```bash id="o1s6dy"
kubectl logs -f <pod-name> <container-name>
```

# Important Notes

* Logs are container-specific.
* Kubernetes captures logs automatically.
* Multi-container pods require specifying container names.
* `kubectl logs` is sufficient for basic troubleshooting.
* Advanced logging solutions are covered separately.

# Key Takeaways

* Kubernetes logs are accessed using `kubectl logs`.
* `-f` streams logs live.
* Multi-container pods require container names in commands.
* Logs are essential for debugging and monitoring applications.
* Kubernetes basic logging is simple and developer-friendly.
* Advanced logging stacks are used in production environments.
